- Технология: ALL (MPI + OpenMP)
- Вариант: 4

## 1. Контекст
Для решения задач сверхбольшой размерности возможностей одного узла может быть недостаточно. Гибридная реализация (MPI + OpenMP) позволяет распределять матрицу между узлами кластера и параллельно обрабатывать данные внутри каждого узла.

## 2. Постановка задачи
Реализовать умножение CRS-матриц, где MPI отвечает за распределение строк матрицы A, а OpenMP ускоряет локальные вычисления внутри каждого процесса.

## 3. Базовый алгоритм
1. **MPI уровень:** Матрица A делится на горизонтальные полосы (диапазоны строк). Матрица B рассылается всем процессам целиком.
2. **OpenMP уровень:** Внутри локального диапазона каждый процесс выполняет двухпроходное вычисление.
3. **Сборка:** Локальные результаты (размер которых заранее неизвестен) собираются на Rank 0 с помощью `MPI_Gatherv`.

## 4. Межпроцессная схема
- **Распределение:** `MPI_Bcast` для передачи размеров и структуры матрицы B.
- **Синхронизация:** `MPI_Gather` для сбора счетчиков элементов и `MPI_Gatherv` для сбора самих данных.
- **Валидация:** Результат `ValidationImpl` на Rank 0 рассылается всем остальным через `MPI_Bcast` для согласованности.

## 5. Внутрипроцессная схема
- Внутри каждого MPI-ранга используется `#pragma omp parallel for` с динамическим планированием.
- Каждый поток имеет собственные буферы-аккумуляторы для предотвращения гонок данных.

## 6. Детали реализации
- **Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`.
- **Сборка результата:** После `Gatherv` на Rank 0 выполняется финальный `Bcast` итоговой матрицы всем процессам, чтобы обеспечить успешное прохождение тестов фреймворка на всех рангах.

```cpp
// File: all/src/ops_all.cpp
if (rank == 0) {
  c.values.resize(total_nnz);
  c.col_ind.resize(total_nnz);
  MPI_Gatherv(flat_values.data(), local_nnz_total, MPI_DOUBLE, c.values.data(), 
              all_nnz_counts.data(), nnz_displs.data(), MPI_DOUBLE, 0, MPI_COMM_WORLD);
  MPI_Gatherv(flat_cols.data(), local_nnz_total, MPI_INT, c.col_ind.data(), 
              all_nnz_counts.data(), nnz_displs.data(), MPI_INT, 0, MPI_COMM_WORLD);
} else {
  MPI_Gatherv(flat_values.data(), local_nnz_total, MPI_DOUBLE, nullptr, 
              nullptr, nullptr, MPI_DOUBLE, 0, MPI_COMM_WORLD);
  MPI_Gatherv(flat_cols.data(), local_nnz_total, MPI_INT, nullptr, 
              nullptr, nullptr, MPI_INT, 0, MPI_COMM_WORLD);
}
BroadcastCRSMatrix(c, 0, a_rows, b_cols);
```

## 7. Проверка корректности
Корректность подтверждена на тестах разной размерности. Гибридная версия корректно обрабатывает случаи, когда количество процессов превышает количество строк (пустые диапазоны).

## 8. Экспериментальная среда
- **Конфигурация:** Тестирование проводилось в режиме `mpiexec -n 2` и `-n 4`.
- **Потоки:** Внутри процесса `PPC_NUM_THREADS` варьировался до 8.

## 10. Выводы
Гибридная версия продемонстрировала способность к масштабированию (время сократилось с 0.126с до 0.052с). Низкая эффективность относительно общего числа workers (7.3%) объясняется ценой коммуникаций: сборка разреженных данных через `Gatherv` и их последующая рассылка (`Bcast`) на одной локальной машине создают значительные накладные расходы.
